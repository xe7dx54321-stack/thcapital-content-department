## 1. 文档目的

这份文档的作用，是把我们前面已经讨论过的市场内容系统规划，正式收束成一份**可直接进入执行阶段的施工表**。

它要回答的不是“这个系统大概长什么样”，而是：

1. 一共分几步施工
2. 每一步具体干什么
3. 每一步的输入、产出、依赖、验收标准是什么
4. 哪些事情先做，哪些事情后做

## 2. 施工总原则

在正式施工前，先把几个原则钉死，避免后面越做越偏：

### 2.1 先做内容主链路，不先做复杂自动化

优先把下面这条链跑顺：

> 热点发现 → Top 5 选题建议 → 人工确认 → 多平台 Draft Pack → 打磨 → 入库 / 待发布 → 复盘

在这条链没有跑顺前，不要把主要精力放到：

- 全自动发稿
- 全平台同时接入
- 复杂增长黑盒
- 过重的 SEO / CRO / 广告体系

### 2.2 先单 Agent 跑通，再拆角色

第一阶段不追求多 agent 编排炫技，而是优先让一个主 agent 先把流程跑顺。

推荐先以 `market-editor` 为主执行单元。

后续再按瓶颈拆分：

- `market-scout`
- `topic-planner`
- `content-writer`
- `publish-ops`
- `content-analyst`

### 2.3 人类始终保留两项关键控制权

1. **选题确认权**
2. **最终发布权**

系统可以提高效率，但不应该在需求尚未稳定前直接越过人类做品牌表达决策。

### 2.4 先把自有 TH Skill 写出来，再考虑大规模装外部 Skill

外部 skill 是材料，不是答案。

真正要落地的，是同行资本自己的市场内容 skill 体系：

- `th-market-brand-context`
- `th-topic-radar`
- `th-source-capture-and-citation`
- `th-draft-pack`
- `th-content-polish`
- `th-content-repurpose`
- `th-publish-ops`
- `th-content-review`

## 3. 施工边界

## 3.1 本轮必须完成的范围

本轮施工明确聚焦：

1. 品牌上下文层
2. 选题雷达层
3. 多平台 Draft Pack 层
4. 文本打磨层
5. 发布队列与最小复盘层

## 3.2 本轮明确不做的范围

以下内容先不进入第一轮主施工：

1. 自动发全平台
2. 复杂广告投放系统
3. 完整 CRO 体系
4. 邮件营销自动化体系
5. 复杂 SEO 程序化页面体系
6. 重度 revops / retention / pricing 协同系统
7. 高风险非官方实验接口的正式接入

## 4. 施工总览

建议本轮施工拆成 **8 步**。

| 步骤 | 名称 | 目标 |
| --- | --- | --- |
| Step 1 | 冻结业务边界与对象模型 | 把系统最小业务闭环和对象定义清楚 |
| Step 2 | 建品牌上下文层 | 让系统知道“同行资本是谁、对谁说话、怎么说” |
| Step 3 | 建 watchlist 与素材入口层 | 把热点监控与原始素材输入层搭起来 |
| Step 4 | 建 Topic Radar 与 Top 5 建议单 | 跑通“发现 → 归并 → 评分 → 建议” |
| Step 5 | 建人工确认与任务推进机制 | 把创始人选择后的推进链条接起来 |
| Step 6 | 建 Draft Pack 生成层 | 让一个选题能落成多平台草稿包 |
| Step 7 | 建内容打磨与成品化层 | 让内容更像同行资本，并具备基础成品感 |
| Step 8 | 建发布队列与复盘层 | 让内容进入待发布管理，并开始形成经验闭环 |

## 5. 详细施工步骤

## Step 1：冻结业务边界与对象模型

### 5.1 目标

先把系统最小闭环中的对象和状态定义清楚，避免后面每做一步，数据结构和目录结构都反复推翻。

### 5.2 要做什么

1. 明确系统中的核心对象：
   - watchlist source
   - raw source packet
   - topic cluster
   - topic candidate
   - approved topic
   - draft pack
   - publish queue item
   - performance review

2. 明确每个对象至少有哪些字段

3. 明确每个对象之间的流转关系

4. 明确目录结构和命名规范

5. 明确第一阶段的平台优先级：
   - 微信公众号
   - 小红书
   - 知乎
   - X

### 5.3 依赖

- 已有开发计划文档
- 已有 skill 清单与业务映射表

### 5.4 产出物

建议至少形成：

- `对象字典文档`
- `目录结构文档`
- `状态流转文档`

### 5.5 验收标准

达到以下标准才算完成：

1. 不同对象的边界清楚
2. 选题、草稿、发布、复盘不会混在一起
3. 后续写 skill 时，不需要再重新猜系统对象结构

---

## Step 2：建品牌上下文层

### 5.6 目标

让系统知道同行资本的内容到底该怎么写、对谁写、哪些话该说、哪些不能说。

### 5.7 要做什么

1. 先写 `th-market-brand-context` skill

2. 在这个 skill 里明确：
   - 同行资本品牌定位
   - 主要受众
   - 平台定位差异
   - 语气风格
   - 高频表达
   - 禁写项 / 风险项
   - 内容边界
   - 哪些内容值得发，哪些不值得发

3. 形成一个可持续更新的品牌上下文文档

4. 后续所有内容 skill 默认先读取这份上下文

### 5.7A 当前阶段补充约束

本轮 Step 2 只落 **公域品牌上下文层**：

- 私域维护与社群运营由真人负责
- 但公域内容需要具备“自然转化到后续交流”的意识
- 因此 Step 2 需要额外定义 CTA 与敏感边界规则

### 5.8 对应 skill 来源

- `product-marketing-context`
- `content-strategy`

### 5.9 产出物

- `th-market-brand-context/SKILL.md`
- `brand-context.md` 或同类参考文档

### 5.10 验收标准

1. 系统能清楚区分“这是同行资本该说的话”与“这不是同行资本该说的话”
2. 后续写稿时能显著减少风格漂移
3. 人类不需要每次重新解释品牌调性

---

## Step 3：建 watchlist 与素材入口层

### 5.11 目标

把“看哪里、抓什么、如何变成原始素材”这层正式搭起来。

### 5.12 要做什么

1. 建立首版 watchlist：
   - 竞品机构账号
   - 行业媒体账号
   - 关键 KOL / 创始人 / 投资人账号
   - 平台趋势入口

2. 为不同平台建立 source registry

3. 明确哪些平台走什么读取方式：
   - X / Twitter
   - YouTube
   - 微信公众号
   - 网页长文
   - 小红书（如后续需要）

4. 建立原始素材落盘规范：
   - 来源链接
   - 标题
   - 平台
   - 抓取时间
   - 正文 / 转写结果
   - 可引用片段

5. 建立 `th-source-capture-and-citation` skill 雏形

6. 建立一份 **seed 重选策略 + 候选池**
   - 把固定 seeds 与动态 scouting rules 分开
   - 把 source 分成 `A 类正式盘 / B 类战略源 / C 类人工补录`
   - 不再用几个占位 URL 代表整个业务

7. 建立 `th-seed-refresh-and-source-scouting` skill
   - 用来维护动态发现路径
   - 处理“不是固定 URL、但又必须长期做”的找源动作

### 5.12A Step 3 的重要纪律

Step 3 的 watchlist 与 source registry，需要遵守一条硬原则：

> **信息源层面保持中立，不用“主战场 / 邻接战场”给 source 分等级。**

原因：

- 市场真正高热、真正容易爆的题，可能来自主战场，也可能来自邻接领域
- 如果在 source 层就先把邻接方向边缘化，系统会漏掉大量值得做的大题
- 品牌取舍应该发生在 Step 4 的候选题推荐阶段，而不是发生在 Step 3 的信息输入阶段

因此 Step 3 更应该按以下维度管理 source：

- 信号质量
- 热度与增速
- 持续产题能力
- 来源可信度
- 可引用性
- 平台覆盖价值

而不是按“是不是主战场”做前置压制

### 5.13 对应 skill 来源

- `Agent Reach`
- `Youtube-ultimate`
- `baoyu-url-to-markdown`
- `baoyu-youtube-transcript`
- `baoyu-danger-x-to-markdown`
- `baoyu-translate`

### 5.14 产出物

- watchlist 清单
- source registry
- seed 重选策略
- seed 候选池
- 原始素材目录
- `th-source-capture-and-citation/SKILL.md`
- `th-seed-refresh-and-source-scouting/SKILL.md`
- source packet 模板
- 信源抓取与引用落盘规范

### 5.15 验收标准

1. 系统知道每天看哪些源
2. 原始素材能稳定落档
3. 原文链接和引用关系不丢失
4. 海外材料可以进入中文工作流

## Step 4：建 Topic Radar 与 Top 5 建议单

### 5.16 目标

让系统先尊重市场，再体现品牌身份。

把“看到了很多东西”真正转成“今天最值得写什么”。

### 5.17 要做什么

1. 写 `th-topic-radar` skill

2. 在 skill 中明确：
   - 热点聚类规则
   - 市场热度判断
   - 相关性判断
   - 多平台发酵度判断
   - 增长速度判断
   - 持续性判断
   - 可写性判断
   - 差异化空间判断
   - 品牌贴合度判断
   - 主战场 / 邻接题的后道重排逻辑
   - 风险判断

3. 规定 Top 5 建议单模板

3A. 规定 Top 8 → Top 5 的汇报纪律：
   - 最终进入 Top 5 的 5 个必须解释“为什么推荐”
   - 未进入 Top 5 的 3 个也必须保留在板子上
   - 这 3 个要写清楚：
     - 为什么能进最终 Top 8
     - 为什么被放下
     - 是否允许创始人手动捞回
     - 如果捞回，最适合改成什么角度

4. 规定每个建议项必须输出：
   - 题目
   - 一句话判断
   - 为什么值得做
   - 市场潜力
   - 品牌贴合度判断
   - 哪些平台在发酵
   - 原始链接
   - 建议切入角度
   - 适合哪些平台
   - 风险提示

### 5.18 对应 skill 来源

- `content-strategy`
- `marketing-ideas`
- `social-content`
- `Agent Reach`

### 5.19 产出物

- `th-topic-radar/SKILL.md`
- Top 5 建议单模板
- Top 8 → Top 5 board 模板
- 话题簇输出模板

### 5.20 验收标准

1. 每日建议不是素材搬运，而是已经做过判断的建议
2. 创始人能快速看懂并做选择
3. 建议单里保留原始链接，方便核对

## Step 5：建人工确认与任务推进机制

### 5.21 目标

让创始人只做轻量决策，不用亲自重新组织内容任务。

### 5.22 要做什么

1. 定义 topic candidate → approved topic 的流转规则

2. 定义创始人确认后的动作协议，例如：
   - 选第 2 个
   - 第 3 个改成机构视角
   - 第 1 个只做微信和知乎

3. 建立 confirmed topic 的对象结构

4. 明确确认后自动进入 Draft Pack 队列

### 5.23 对应 skill 来源

- 这里主要不是外部 skill，而是系统流程设计
- 会与 `th-topic-radar`、`th-draft-pack` 直接衔接

### 5.24 产出物

- 状态流转规则
- confirmed topic 模板
- 执行触发规则

### 5.25 验收标准

1. 人类确认动作足够轻量
2. 确认后不需要人工重复解释任务
3. 系统知道下一步该产出哪类内容包

## Step 6：建 Draft Pack 生成层

### 5.26 目标

让一个已确认的题目，能自动生成多平台版本，而不是只出一篇长文。

### 5.27 要做什么

1. 写 `th-draft-pack` skill

2. 先定义 Draft Pack 的统一输出结构：
   - 微信版
   - 小红书版
   - 知乎版
   - X 线程版
   - 标题候选
   - 摘要 / 导语
   - 配图建议
   - 引用来源清单
   - 风险备注

3. 为每个平台分别定义稿件规则：
   - 长度
   - 结构
   - 语气
   - 标题特征
   - 信息密度要求

4. 保证不是“一稿四投”，而是按平台重写

### 5.28 对应 skill 来源

- `copywriting`
- `social-content`
- `content-strategy`

### 5.29 产出物

- `th-draft-pack/SKILL.md`
- 平台模板
- Draft Pack 目录结构

### 5.30 验收标准

1. 同一主题的不同平台版本明显不同
2. 仍保留同一套核心判断，不会写散
3. 创始人选题后能快速拿到成稿 / 准成稿

## Step 7：建内容打磨与成品化层

### 5.31 目标

让内容不只是“写出来”，而是“像同行资本写出来”。

### 5.32 要做什么

   - 风格统一

   - 空话
   - 鸡汤
   - 过油表达
   - 套话腔
   - 没有信息量的感叹句

#### B. 再建基础成品化能力

4. 微信稿 HTML 化
5. 封面与配图
6. 小红书图卡化
7. 视情况补信息图能力

1. 成稿明显更像品牌内容，而不是 AI 草稿

3. 小红书版具备基础图卡传播形态

## Step 8：建发布队列与复盘层

### 5.36 目标

让内容从“写完”变成“可发、可追踪、可复盘”。

### 5.37 要做什么

#### A. 发布队列

1. 定义 publish queue item 对象
2. 记录每篇内容当前状态：
   - 待修改
   - 待发布
   - 已发布
   - 暂缓
3. 建立提醒机制
4. 第一阶段先做“提醒人发”，而不是“自动发”

#### B. 最小复盘层

5. 写 `th-content-review` skill 雏形
6. 记录以下信息：
   - 哪个选题被选中
   - 最终发了哪些平台
   - 哪个平台效果更好
   - 哪类标题更容易通过
   - 哪些 source 更容易贡献好题

7. 定义周 / 月复盘模板

### 5.38 对应 skill 来源

- `analytics-tracking`
- `ab-test-setup`
- `social-content`
- `baoyu-post-to-wechat`（后续可作为增强）
- `baoyu-post-to-x`（后续可作为增强）

### 5.39 产出物

- `th-publish-ops/SKILL.md`
- `th-content-review/SKILL.md`
- publish queue 模板

### 5.40 验收标准

1. 每篇内容当前处于什么状态一目了然
2. 系统不再“写完就算结束”
3. 能开始积累题材效果和来源效果数据

## 6. 建议的实际施工顺序

虽然上面拆成了 8 步，但执行时建议按下面顺序推进：

### 第一批：先跑最小业务闭环

1. Step 1 冻结对象模型
2. Step 2 品牌上下文层
3. Step 4 Topic Radar
4. Step 5 人工确认推进机制
5. Step 6 Draft Pack
6. Step 7 文本打磨

### 第二批：补素材与成品化

7. Step 3 素材入口层
8. Step 7 基础成品化部分（配图 / 图卡 / HTML）

### 第三批：补发布与复盘

9. Step 8 发布队列
10. Step 8 最小复盘

原因很简单：

> 如果“品牌上下文 → 选题建议 → 人工确认 → Draft Pack → 打磨”这条线没跑通，其他所有增强能力都很容易变成空转。

## 8. 施工完成的最低标准

如果要说“第一阶段市场内容系统基本做完了”，我建议最低标准是：

1. 系统每天能输出 Top 5 选题建议
2. 创始人选题后，系统能自动生成多平台 Draft Pack
3. 成稿经过打磨后，有明显品牌感，不像通用 AI 草稿
4. 微信稿和小红书稿都能形成基础成品
5. 所有内容都能进入发布队列管理
6. 发布后有最小复盘闭环

只要做到这 6 条，就说明这套系统已经从“概念”进入“能用”的阶段。
