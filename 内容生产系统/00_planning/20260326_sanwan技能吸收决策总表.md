## 2. 我们的判断框架

只有同时满足下面至少 2 条，才值得吸收：

1. 能补我们当前真实缺口
2. 能提高信息获取或内容生产效率
3. 能让前台协同更顺
4. 能改善复盘和进化能力
5. 不会破坏我们现有业务边界与隔离原则

反过来，以下情况一律不吸收：

- 只是“看起来很炫”，但和主链路无关
- 会把我们的平台策略带偏
- 会把前台流程复杂化
- 会把自动化权限开得过大
- 会让外部 skill 直接替代我们自己的业务 knowhow

---

## 3. 五层能力栈

我们把所有有价值的技能统一映射到 5 层：

1. **组织层**：多 agent、前后台协同、审批、任务流
2. **情报层**：热点发现、反爬抓取、正文净化、多源研究
3. **生产层**：按平台生成内容、改写、排版、人性化
4. **运营层**：发布协同、前台展示、工作台、人工审核
5. **进化层**：复盘、总结、能力演化

这 5 层里，当前最重要的顺序是：

> **情报层 > 生产层 > 组织层 > 运营层 > 进化层**

不是因为进化不重要，而是前面没跑通，后面无从进化。

### A 档一句话结论

A 档不是“装上就用”的软件清单，而是：

> **我们内容工厂最核心的 17 个能力部件。**

少了它们，系统就会变成：

- 只会写，不会找
- 只会找，不会洗
- 只会出稿，不会平台化
- 只会跑流程，不会进化

## 6. 技能包怎么处理

技能包最容易让人犯两个错误：

1. 觉得“整包装上 = 拥有能力”
2. 觉得“别人这样搭，我们就照抄”

我们对技能包的处理原则非常明确：

### 6.1 内容创作者包

- **吸收**：公众号 / 人性化 / 图片 / 博客这一组“内容生产链”
- **不吸收**：它对“全平台一条龙”的轻量创作者视角
- **我们的做法**：保留“组合拳”思想，但坚持“主阵地 + 平台束 + 差异化表达”

### 6.2 出海运营官包

- **吸收**：X + SEO + 搜索 + 深研的联动
- **不吸收**：把海外增长逻辑生搬进国内内容工厂
- **我们的做法**：只把它当作 `X` 和 `SEO` 的增强思路

### 6.3 三万同款团队

- **吸收**：多 agent 协作的架构抽象
- **不吸收**：角色命名、业务分工、全部技能全集
- **我们的做法**：用我们的业务链定义角色，而不是用它的角色反推我们的流程

## 7. 真正的“狠决策”

### 7.1 立即执行的

下面这些，不是“可以研究一下”，而是**必须吸收进我们自己的能力设计**：

- 多 agent 组织法
- 搜索 / 深研 / 抓取 / 净化四件套
- 微信 / 小红书 / X 的平台原生逻辑
- 去 AI 味与人格一致性
- 飞书前台协同
- 运行历史驱动的能力进化

### 7.2 暂不直接施工，但必须纳入设计预留位的

- SEO 结构能力
- 视频内容转文字能力
- 视觉物料自动初稿
- 多维表项目化管理
- 自动化工作流重试与人工审核链

### 7.3 明确禁止的

- 禁止“整包安装后照搬”
- 禁止把自动发布当作当前主目标
- 禁止让浏览器自动化承担主生产链
- 禁止把 SEO 逻辑硬套给所有平台
- 禁止把外部 skill 的 prompt 当作我们的业务 knowhow

## 9. 最终结论

如果只用一句话概括这次尽调的结果：

> **Sanwan 对我们最有价值的，不是“内容写作”，而是“多 agent 协同 + 反爬抓取 + 平台原生运营 + 自我进化”这四件事。**

我们后续真正要做的，不是“把它的技能搬到我们这里”。

而是：

> **把这些能力的核心逻辑拆出来，塞进我们自己的内容工厂体系里，让它们服务于同行资本，而不是反过来定义同行资本。**
