# 同行资本市场内容系统｜平台覆盖战略与分发矩阵建议

## 1. 本次修订结论

自 `2026-03-26` 起，内容工厂 Phase 1 的正式支持平台统一升级为 7 个：

- 微信公众号
- 小红书
- 知乎
- X
- B站专栏
- 今日头条
- 百家号

这次修订只解决两件事：

1. **能力层**：内容工厂、模板、runbook、导出能力都要原生支持这 7 个平台。
2. **策略层**：这 7 个平台都进入标准打法，但**不意味着每个题都必须 7 平台齐发**。

核心原则只有一句：

> **Phase 1 是“7 平台标准支持”，不是“7 平台机械群发”。**

---

## 2. 总体平台战略

对于“同行资本｜AI Agent 前沿机会雷达”这条线，平台必须分工，而不是平铺。

建议采用以下结构：

### 2.1 主阵地

1. **微信公众号**
   - 内容资产主库
   - 信任沉淀主场
   - 私域转化落点

2. **小红书**
   - 公域破圈引擎
   - 冷启动增长主场
   - 标题/钩子/低门槛表达测试场

3. **知乎**
   - 搜索承接主阵地
   - 专业问题回答主场
   - 长尾认知资产沉淀层

### 2.2 辅助分发层

4. **X**
   - 全球信号雷达
   - 海外事件快评窗口
   - build in public 的轻量表达场

5. **B站专栏**
   - 年轻 builder / 技术社区补位
   - 更适合“案例拆解 / workflow 拆解 / 工具实操 / 长解释”
   - 不是视频优先打法，而是**专栏深度补位**

6. **今日头条**
   - 泛流量铺量实验层
   - 标题包装 / 大众表达 / 破圈实验场
   - 用来验证哪些题在更广泛用户侧也能成立

### 2.3 搜索镜像层

7. **百家号**
   - 百度搜索镜像层
   - SEO 与关键词承接层
   - 适合沉淀解释型、问答型、框架型内容

---

## 3. 为什么 7 平台都进 Phase 1

这次不是简单“多加几个平台”，而是为了把不同增长目标拆开承接：

- **公众号** 解决内容资产和信任沉淀
- **小红书** 解决冷启动曝光和破圈
- **知乎** 解决问题搜索与专业承接
- **X** 解决全球前沿信号和海外语境
- **B站专栏** 解决 builder 社区和教程/案例深读
- **今日头条** 解决泛流量分发与包装实验
- **百家号** 解决百度收录与 SEO 长尾

如果只做 4 平台，系统会出现三个缺口：

1. **缺社区深度补位**
   - 公众号和知乎够“稳”，但缺一个更靠近年轻 builder 社区、对案例拆解更友好的阵地。
   - 这个角色由 **B站专栏** 承担。

2. **缺大众表达实验场**
   - 小红书擅长低门槛表达，但不等于能覆盖更广泛的大众流量测试。
   - 这个角色由 **今日头条** 承担。

3. **缺百度搜索镜像**
   - 知乎的搜索属性强，但百度外延面还不够。
   - 这个角色由 **百家号** 承担。

---

## 4. 但为什么不能每篇都 7 平台齐发

因为平台分工不同，受众状态不同，内容任务也不同。

如果把每个题都硬拆成 7 平台，会带来 4 个问题：

1. 生产负担快速失控
2. 内容开始同质化、机械改写
3. 平台特征被抹平
4. 审稿与发布链路爆炸，反而拖慢节奏

所以真正的执行原则是：

> **每个题进入 Phase 1 平台池，但只选择最合适的“平台束（platform bundle）”来发。**

## 5. Phase 1 的平台束设计

### 5.1 标准平台束

#### A. 深度认知束

适用：趋势解释、案例拆解、判断型内容

- `wechat`
- `zhihu`
- `baijiahao`

#### B. 破圈传播束

适用：热点快拆、误区纠偏、轻结论内容

- `xiaohongshu`
- `toutiao`
- `x`

#### C. Builder 社区束

适用：workflow、skill、工具、案例方法论

- `bilibili`
- `zhihu`
- `wechat`

#### D. 全信号快反束

适用：海外模型更新、产品发布、事件快评

- `x`
- `xiaohongshu`
- `wechat`

#### E. 搜索沉淀束

适用：解释型、问答型、可长期被检索的话题

- `zhihu`
- `baijiahao`
- `wechat`

### 5.2 执行纪律

- 不是每个题都必须用现成平台束。
- 可以按题目微调，但必须解释为什么选这些平台。
- 最低要求是：
  - 有一个主阵地版本
  - 有一个分发或增长版本
  - 如具备 SEO 价值，则补一个搜索镜像版本

## 7. 按内容类型选择平台

### 7.1 热点快拆

优先平台：

- 小红书
- X
- 微信公众号
- 今日头条

可选补位：

- 知乎

### 7.2 趋势解释 / 判断型内容

优先平台：

- 微信公众号
- 知乎
- 百家号

可选补位：

- X

### 7.3 workflow / skill / builder 案例

优先平台：

- B站专栏
- 微信公众号
- 知乎

可选补位：

- 小红书

### 7.4 搜索友好型 evergreen 内容

优先平台：

- 知乎
- 百家号
- 微信公众号

### 7.5 大众表达 / 标题测试型内容

优先平台：

- 小红书
- 今日头条

## 8. 平台间怎么打配合

### 8.1 主阵地逻辑

- 真正的“完整判断”优先沉淀到 **公众号**
- 真正的“问题回答”优先沉淀到 **知乎**

### 8.2 破圈逻辑

- 小红书负责把复杂内容翻译成低门槛入口
- 今日头条负责测试更大众化、更强包装的版本

### 8.3 社区逻辑

- B站专栏负责把 workflow / skill / builder 实操讲透
- 让内容更容易被懂工具、爱折腾、爱拆解的人群接受

### 8.4 搜索逻辑

- 知乎和百家号形成双搜索面
- 知乎偏“问题搜索”
- 百家号偏“百度收录与关键词镜像”

### 8.5 信号逻辑

- X 负责最前沿事件快评与一手语境
- 但不强求它承担国内主要转化

---

## 9. Phase 1 的执行口径

### 9.1 系统能力口径

Phase 1 必须原生支持：

- `approved_topic` 可指定 7 平台任意组合
- `draft_pack` 可生成 7 平台稿件
- `docx export` 可导出 7 平台稿件
- publish queue 能识别 7 平台路径

### 9.2 运营口径

Phase 1 不要求：

- 7 平台全部接入 API
- 7 平台全部自动发布
- 7 平台每天都必须有产出

Phase 1 要求的是：

- 这 7 平台都进入正式战术盘
- 选题评估时可合法被选中
- 草稿生产链能无缝覆盖

---

## 10. 当前建议的优先发稿顺序

如果同一个题同时具备多个平台版本，建议优先级如下：

1. 微信公众号
2. 小红书
3. 知乎
4. B站专栏
5. X
6. 今日头条
7. 百家号

解释：

- 前三者是主阵地
- B站专栏是本轮新增重点补位
- X 偏信号与外部可见性
- 头条和百家号更像增长实验与搜索镜像

---

## 11. 一句话收口

新版平台战略不是：

> “再多做 3 个平台。”

而是：

> **把内容工厂正式升级成一个 7 平台分层系统：公众号沉淀、小红书破圈、知乎承接、X 试信号、B站补社区、头条做实验、百家号做 SEO。**

真正的执行关键不是“多发”，而是：

> **每个题都选对平台束，每个平台都承担自己最合适的任务。**
