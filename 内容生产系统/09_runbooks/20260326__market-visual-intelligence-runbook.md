# 同行资本市场内容系统｜图片理解与取图执行 Runbook

## 1. 目标

在真实接入生产前，把内容工厂补齐两段图片能力：

1. `图片理解`
   - 在 source intake 阶段识别图片里承载的关键信息
   - 判断图片和正文之间的配合关系
   - 为后续重构内容时提供“哪里该放图、放什么图”的依据

2. `取图执行`
   - 能对原始推文、原始公告、PDF 标题页、产品页、Repo 页等做截图
   - 能给出取图优先级和降级路径
   - 能把图片交接到 draft pack / publish queue，而不是只停留在“建议配图”

---

## 2. 能力落点

本轮把图片能力接到三层：

### A. `source packet`

新增 `Visual Intelligence` 段，负责记录：

- 图片 / 缩略图 / 榜单图 / 页面截图候选
- 原始图像可能承担的 job
- 可复用的证据图位置
- 后续正文里适合插图的建议

### B. `draft pack`

新增 `inline-visual-plan.md`，负责把图文结构显式化：

- 哪个平台需要几张图
- 每张图放在文章哪个位置
- 这张图的 job 是什么
- 优先抓原始截图还是做解释图

### C. `publish queue`

新增 `visual-asset-sourcing.md` 进入 handoff 资产：

- 原始截图候选
- 抓图命令
- 建议输出路径
- 抓不到时的 fallback

---

## 3. 图像优先级规则

统一按以下顺序取图：

1. `原始证据图`
   - 原始推文截图
   - 原始公告页标题区
   - 原始 PDF 首页标题
   - 产品页 / Repo 页原始截图

2. `官方现成资产`
   - 官方产品图
   - 官方缩略图
   - 官方 hero image

3. `解释型辅助图`
   - 结构图
   - 流程图
   - 变量卡片

4. `外部补图`
   - 媒体图
   - 搜索到的对象图
   - 当前默认检索源为 `Wikimedia Commons` 开放许可素材
   - 只在与主题实体 / 原始对象足够相关时进入成稿预览
   - 默认只补 `1` 张，避免外部图反客为主

5. `AI 生成图`
   - 只在没有合适原始图，且图的 job 是解释结构而不是证明事实时使用
   - 当前通过 `OpenAI Images API` 调用；若环境缺少 `OPENAI_API_KEY`，自动退回本地结构解释图

### 3.1 取图决策树

执行时不要机械地“统一截图”。按下面顺序判断：

1. `先问这张图要完成什么 job`
   - 如果要证明事件真实发生，优先找原始证据截图
   - 如果要证明对象 / 品牌 / 产品长什么样，优先找官方资产
   - 如果要帮助读者理解结构，才考虑解释图

2. `如果来源本身是官方图片直链 / 官方海报 / 官方缩略图`
   - 直接下载原图文件
   - 不要把浏览器打开图片后的黑底页面再截图

3. `如果来源是官方页面`
   - 优先截标题区、hero 区、对象区、repo 首屏、PDF 首屏
   - 不要无差别整页截图

4. `如果手里只有第三方媒体文章`
   - 默认不自动截图文章页
   - 优先回找原始帖子、官方公告、repo、产品页、PDF、官方活动海报
   - 第三方文章中的图，只有在确认无水印、无版权风险、且服务正文表达时才考虑复用

5. `如果官方资产只能证明对象，不能证明事件`
   - 正文里要补一张证据型截图
   - 不允许只用抽象品牌图去证明“事件发生”

6. `如果外部检索只搜到“字面相关但业务无关”的图`
   - 直接放弃外部补图
   - 退回解释型图 / AI 解释图
   - 宁可少一张图，也不要把不服务正文的图塞进去

---

## 4. 平台插图规则

### `wechat`

- 头图之外，正文至少要有 `2-4` 个视觉停顿点
- 第一张正文图优先做“原始证据锚点”
- 结构图 / 对比图应放在变量段而不是导语段

### `zhihu`

- 优先解释型图，而不是营销封面
- 最适合插在“背景讲清后、判断展开前”

### `xiaohongshu`

- 图本身就是内容主体
- 优先做“封面卡 + 解释卡 + 变量卡”

### `bilibili`

- 更像 build log 的可视化辅助
- 适合对象截图 + 拆解标签

### `x`

- 图片只做加分项，不代替首条文字
- 若有强原始证据图，可以作为 thread 中第二条或第三条

---

## 5. 截图执行规则

- 默认优先截“标题区 / hero 区 / 对象区”，避免整页满屏无重点
- 若对象是 `X` 推文，优先保留原帖正文、作者、时间与互动区附近
- 若对象是 `PDF`，优先截首页标题 + 发布主体
- 若对象是 `GitHub`，优先截 repo header、stars、readme 首屏
- 若对象是 `YouTube / B站`，优先截视频标题区或使用官方缩略图
- 若来源本身就是 `官方图片直链 / 官方活动海报 / 官方 CDN 图片`，优先下载原图文件，而不是截图图片直链页面
- 若来源是第三方公众号 / 媒体文章页，默认禁止整页截图直接入正文；优先回找原始源、官方资产或替代安全图
- 若截图失败或来源不安全，自动降级顺序应为：官方 hero / 缩略图 → repo / 产品页安全截图 → 开放许可外部补图（最多 1 张）→ AI / structured explainer

---

## 5.1 成稿预览准入规则

最终进入稿件池 / 平台仿真的图片，只能来自：

1. 原始证据截图
2. 官方资产 / 官方缩略图 / 官方活动海报
3. 经过明确批准的外部安全图

以下内容默认不得进入最终成稿预览：

- 内部占位图
- 内部解释卡
- 取图说明卡
- 业务流程图
- 黑底图片页截图

这些东西可以帮助内部生产，但不能污染最终成稿页。

---

## 6. 交付标准

生产前至少满足：

1. source packet 能输出图片相关理解
2. draft pack 能输出图文插槽规划
3. publish queue 能带出取图执行清单
4. 系统里有一个稳定可调用的网页截图脚本

只有这样，内容工厂才不至于停留在“知道应该配图，但没人知道该配什么图、怎么取图”。
