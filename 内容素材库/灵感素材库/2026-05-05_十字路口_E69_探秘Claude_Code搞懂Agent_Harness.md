那我们再回到聊这个Agent Harness，当我们聊Agent Harness的时候，一般来说新路你会把它再拆成几个部分吗？我会倾向你把它拆成三个部分，就是第一层的是给模型去提供这个它的执行能力的一个层，简单像CLI也好，包括你去习达马去注册的那些工具，包括从MCP去扩展的工具，这些统一是给模型提供的Action的能力。那第二层我会更偏向于类似于叫做上下文，就是状态这些层面。比如说你的模型你要去工作，它是不是需要有它的System Prompt，它有它的Skills，然后它有它的Memory，包括这个模型的上下文窗口不是无限的，很多的任务它其实会超过模型的上下文窗口，它是不是会要有一些offload的，然后下一轮相当于其实，你以为还是那个Agent Harness在工作，其实已经是一个新的模型窗口，在装上一些处置化上下文，让它继续上一个窗口里再进行的工作，其实有点像一个新的Agent的。那它怎么接上上一个Agent交接给它的工作？这里面会有很多的上下文和状态的问题。第二层我觉得它被归为上下文的环境层也更好。那第三层把它归为就是说对Agent的这个上层的管理，这个管理也会分两个方面。一个是如果你今天面对的不是这一个Agent，你是面对100个Agent的，就像你公司有一百号员工，你怎么去组织他们协调起来，按照某种组织架构，某种协作方式和传递顺序，去把一个复杂的事情给解决掉。第二层就是说你针对这100个人，对吧，它每个人要不同的岗位的访问权限，其实也会有很多这种关于权限治理，关于信息的提供和可理，有很多这种治理的方面问题。

所以这三层一个是执行能力层，然后一个是上下文的环境层，还有一个是治理编排就管理这一层。那我们要不要用一个例子来串一串这三层，就一个具体的Agent的任务，它的背后Harness的三层是怎么发挥作用的？之前有一个证明的例子，就是说就是用两周时间多协调达连的Agent从0到1构建了一个C编译器，它通过这个非常多Agent交替完成任务，大家就把这个任务给完成了。那我觉得这是一个很好的case。那首先我们说，首先Agent的最底层是一个模型，对吧，模型就像大脑和心脏一样，它要去驱动整个任务的完成。那它要有Action能力，所以说你至少要给它配置类似于有文件的增删读写搜索等等，这样对应的工具的能力就是Action能力，就是我说的Harness的第一层，你给模型提供的能力。那么模型才有机会去写代码，去创建文件，修改它前面创建过的文件。那第二层就是上下文状态这一层，OK，那我模型是不是要知道我现在是在一个什么样的环境中去工作，然后我的路径是什么，我这上面装有的那些依赖，对吧，然后我已经写的文件芽的结构是什么，对吧，然后包括我的Gate信息，以及它的模型上下文窗口非常满。因为我们知道一个CBN编译器是一个很大的工程量，它不可能在一个模型的上下文窗口中直接去做完而不装满。它要有些上下文的卸载策略，包括我们刚说的上下文窗口满以后，下一个Agent再去工作，它要去重新去读一下我当年环境信息是什么，前面已经写的代码写到哪了，接下来我要进行什么。在我这一轮的成名周期里做到某个阶段，我会写类似的文档去委托我当天的上下文状态，再交给下一个Agent，这样一次的接力最后去把整个CBN编译器写完。它不是一个单纯的Action能力层就能转成的，它还是需要上下文环境的，对。然后第三层就是编排和治理层，如何去知道这些Agent之间去做一个传递和委托，包括我的写代码Agent和测试Agent之间的关系，包括这个任务中难道一定每个环节都是串形的关系吗，我只能有一个Agent接着一个Agent去完成吗，我是不是有很多某块我是可以并行的去写的，可以同一时间有多Agent去做这个文人？他们之间的协调关系是什么？就是这会涉及到编排的一个问题，以及就是说那不同的Agent它的权限怎么样，我去做测试的Agent是不是应该有测试环境，包括测试工具它对应的权限。它应该在测试的时候对着我的这颗代码，它一边测一边修改说，我测不过我把那儿修改一下就好了，然后它一直在Hack这种法子，对吧？所以说这里面会有很多的这种全线的治理，包括上层的这种协调关系的编排的一些问题，这属于第三层。

要不这个也请你介绍一下，你们在做的这一层具体是什么？就是这是你们创业的一个项目对吧？我们在做这一层，我要刚刚说的Agent的三个层面，我们去提供了一个完整的工具链。首先我们最底层是一个叫Komputer的工具链，就是它是取Komputer C的那个字母变成K，叫Komputer。它是一个我们在数据结构上实现的一台Unix计算机，就是你可以理解为我们在内存中实现的一台虚拟的计算机。它给我们像小龙虾，包括Claude Code等这种，下一代的一种主动式的Agent，它有它的自底带性成长性，我们要给这种Agent配一个它的执行环境，就是让它在那个环境里去生活和工作。然后这是我们提出的，我们认为Unix的计算机可能就是对它最好的居住和生活的环境。我们做的第一层是这个，对。然后我们上层会有一个叫K Runtime的东西，它是一个Agent Runtime这一层，我们会相当于提供给开发者去开发Agent的对象的非常多的接口，语法糖上的一个封装。除了这一层，我们还会有其他很多跟大家正交的层面。比如说像K Watch，对吧，这是我的观测。比如说过去半个月我的Agent他们分别在任务完成在什么情况下遇到卡点，对吧？它这样就可以知道我去观测，这不是我的LLM有设计好，还是我的Skills没有提供到位，还是说我用的模型不太行，OK，一个时候观测层就很重要了。以及我们基于观测层的话，可以对数据做导出，导出以后我们会有一个叫KRL的一个工具链，你可以把这个数据拿来强化学习去训练你的模型，或者说你说OK，我不想练我自己的模型，我就想用一个标准的模型的API去请求那个计划，我们也可以对你做这个上下文层面，比如说关于Memory，关于Skills，关于过去的一些经验的一提取和自底带的一个，就是你可以理解为成功人版的强化学习。

听起来其实配工具这一层很简单，但是这一层，有你认为比较容易采的坑吗？它很多时候跟我们的全线，包括Agent的角色是相关的。比如说说我某一个Agent就是负责去对这个代码库做一遍explorer对吧，然后这个时候我应该只给它配就是没有副作用的工具，比如说所有关于写的命令，就是包括杀啊改的能力，包括有一些你是不能让它操作Browser，或者就是你配给它以后你是限制它访问很多网域的，就是这样的话会有很多的这种中间的Tradoff，就是你的Agent的工具链的设计，要紧密的跟Agent的角色绑定去Limit它的调它的底层，对这台Unix的计算机的操作能力。

那我们刚才说到这个第二层，上下文与状态层这环境层的时候，Memory就记忆是非常重要的。然后现在也可以看到有不同的记忆，不管是这个模型厂原生也有一些记忆的功能，对吧，然后也有三方的一些开源的一些插件，它会给记忆的解决方案，然后可不可以给大家讲一讲，现在你认为Agent Harness的Memory这一层有哪些就是主要的解决方案？

Memory这一层的发展呢，目前还处于一个很早期的阶段，我其实把它处理的分为就是规则式的，半规则式的和完全模型驱动式的。那目前落在使用上来看的话，相对规则式的和半规则式的我也多吧。完全规则式的就是像很多基于这种知识图谱和相量搜索，他们所构成的就是把最后你的这个信息在最后抽象中不同节点的表示，然后他们之间在做关联，然后检索和扩展。那这种方式是非常结构化信息规则式做法的，当然我个人其实不太喜欢这种做法。然后我更喜欢的是像后者这种半结构化的，就是可能我底层是一个Unix Files，我就是通过大量的Markdown的方式来存出这种资料信息，其实Claude Code也是这样做的，省下里面也是这样做的。它可以通过一些类似其他的空间关系，比如说像迷宫这种走势，这是最近生化危机的那个女主最近做的特别红的那个记忆项目，这种其实就是底层找一种大家都公认的非常清晰的，稳当稳健的方式，我有房间和空间位置关系，就是人也很容易懂，这个LLM也很容易懂的一种底层的，半结构化的这种方式的信息。然后它的更新也是半规则式的，就基本上是通过Agent来做一个更新，而不是完全Rubes的，一块一下就最后得到一个什么信息，再在知识图上做一个什么样的推理。我觉得那个是叫完全规则式完全Rubes的，半规则式的做法其实就是用Agent来在这里边去，你把这个流程的规则告诉Agent，但是整个分析和过程是由Agent去跑的，我现在要找什么信息，完全是一个后台的Agent，可能用一个更便宜的小模型去跑一遍。那更新其实也是这样的，像Claude过程中也有这种类似于做梦的机制，就是隔一天让它跑一下最近几个section的对话中，它提取下有用信息是可以对过去的记忆做这种更新，纠正和合并的。然后包括这一块有一个开源项目叫MEMU也做得很好。MEMU，对MEMU是一个开源项目，它其实也非常去拥抱Unix Files，加上Agent去驱动维护更新和查询的整个方式，其实Claude Code里边也是这样设计的。

所以我们说Memory这一层，现在的主流方法主要有三类。第一类是完全结构化的，用知识图谱加向量搜索，这种方法信息存储很结构化，但新路个人不太喜欢。第二类是半结构化的，用Unix Files加Markdown再加上Agent驱动更新，Claude Code和小龙虾都是这么做的。第三类是模型内化，完全靠模型来管理记忆，但新路认为离真正生产落地还需要约三年时间。

我们再聊聊Harness的共识与非共识。共识是"CLI is all you need"，现在已经成了行业共识，大家都在做CLI。但非共识是大部分23年到25年诞生的主流开源Agent框架仍然在用PromptPipe加Node Graph的老路线。Unix从1971年就存在，LLM预训练语料中Linux命令有数十亿条sample；MCP提出才两年，预训练占比不到0.1%——这解释了为什么CLI工具的任务完成率比MCP高。

Claude Code源码泄露让我们看到了他们在上下文管理上做了多少工程工作。上下文压缩策略上，何时删除工具输出、窗口满到什么阈值开始交接、下一个Agent初始化时加载什么，这些都有精细设计。Fork Agent机制也很精妙：每轮结束触发turn stop hook，Fork一个Agent复用KV cache做记忆更新。记忆文件格式和skill遵循同一套哲学：前三行YAML，先读description而非全文。

好的Harness设计原则是：好的context space加上好的action space再加上less prompt control。不好的做法是随意裁剪上下文导致KV cache频繁失效，用Prompt Graph硬控每一步决策让模型越强越被束缚。好的Harness要跟模型的运行逻辑自洽，跟模型未来能力的进步方向正交。

Anthropic从25年初率先从问答模型转向Agent模型训练，领先其他厂商半年。新路看好三个创业方向：第一是Agent Harness工具链，他自己在做；第二是Agent组网，不是给Agent发IM/Mail，而是混合云端/端侧的全设备组网，需要高通量上下文交换；第三是Agent模型集约训练和推理基础设施，类似Tinkercad的方案，集约化高效训练加LoRA热插拔推理。

新路最后分享了一个暴论：以后很多的公司其实更多意义上是一种理财产品，公司不是有人组成的，以后就是0人公司。他从来不觉得1人公司是本地的事情，真正make sense是0人公司。YoYo Agent这个项目就体现了这个概念，它的创作者把它做出来之后，就丢进汪洋大海了，不再改它一行代码，也不再给它一分钱，它要自己去进化，自己去想办法搞到钱来给自己买token。
