### Harness 三层拆解

那我们再回到聊这个 Agent Harness，当我们聊 Agent Harness 的时候，一般来说新路你会把它再拆成几个部分吗？

新璐：我会倾向把它拆成三个部分。

第一层的是给模型去提供它的执行能力的一个层，简单像 CLI 也好，包括你去习得马去注册的那些工具，包括从 MCP 去扩展的工具，这些统一是给模型提供的 Action 的能力。

第二层我会更偏向于类似于叫做上下文层，就是状态这些层面，比如说你的模型你要去工作，它是不是需要有它的 System Prompt，它有它的 Skills，然后它有它的 Memory，包括这个模型的上下文窗口不是无限的，很多的任务它其实会超过模型的上下文窗口，它是不是会要有一些 offload 的，然后下一轮相当于其实你以为还是那个 Agent Harness 在工作，其实已经是一个新的模型窗口在装上一些处置化上下文让它继续上一个窗口里再进行的工作。其实有点像一个新的 Agent 的，那它怎么接上上一个 Agent 交接给它的工作，这里会有很多的上下文和状态的问题。第二层觉得它被归为上下文的环境层也更好。

那第三层把它归为是说对 Agent 的这个上层的管理，这个管理也会分两个方面。

一个是如果你今天面对的不是单个 Agent，你是面对 100 个 Agent 的，就像你公司有一百号员工，你怎么去组织他们协调起来，按照某种组织架构，某种协作方式和传递顺序去把一个复杂的事情给解决掉。

第二层就是说你针对这一百个人对吧，它每个人要有不同的岗位的访问权限，其实也会有很多这种关于权限治理，关于信息的提供和可理，有许多这种治理的方面的问题。

所以这三层一个是执行能力层，然后一个是上下文的环境层，还有一个是治理编排就管理这一层。

那我们要不要用一个例子来串一串这三层，就一个具体的 Agent 的任务，它的背后 Harness 的三层是怎么发挥作用的？

新璐：之前有一个证明的例子，就是说用两周去协调达练的 Agent 从 0 到 1 构建了一个 C 编译器，它通过这个非常多 Agent 交替完成任务，大家就把这个任务给完成了。那我觉得这是一个很好的 case。

首先我们说首先 Agent 的最底层是一个模型对吧，模型就像大脑和心脏一样，它要去驱动整个任务的完成。那它要有 Action 能力，所以你至少要给它配置类似于有文件的增删读写搜索等等，这样对应的工具的能力就是 Action 能力，就是我说的 Harness 的第一层，你给模型提供的能力。那么模型才有机会去写代码，去创建文件，修改它前面创建过的文件。

那第二层就是创下文状态这一层。OK 那我模型是不是要知道我现在在一个什么样的环境中去工作，然后我的路径是什么，我这上面装的那些依赖对吧，然后我已经写的文件牙的结构是什么对吧，然后包括我的 Context 信息，以及它的模型上下文窗口非常满，因为我们知道一个 C 编译器是一个很大的工程量，它不可能在一个模型的上下文窗口中直接去做完而不装满。它要有些上下文的卸载策略，包括我们刚说的上下文窗口满以后，下一个 Agent 再去工作，它要去重新去读一下我当年环境信息是什么，前面已经写的代码写到哪了，接下来我要进行什么。在我这一轮的成名周期里做到某个阶段，我会写类似的文档去委托我当天的上下文状态，再交给下一个 Agent，这样一次的接力最后去把整个 C 编译器写完。它不是一个单纯的 Action 能力层就能转成的，它还是需要上下文环境的。

然后第三层就是编排和治理层，如何去知道这些 Agent 之间去做一个传递和委托，包括我的写代码 Agent 和测试 Agent 之间的关系，包括这个任务中难道一定每个环节都是串形的关系吗，我只能有一个 Agent 接着一个 Agent 去完成吗？我是不是有很多某块我是可以并行的去写的，可以同一时间有多 Agent 去做这个文人，他们之间的协调关系是什么，就是这会涉及到编排的一个问题。以及就是说那不同的 Agent 它的权限怎么样，我去做测试的 Agent 是不是应该有测试环境，包括测试工具它对应的权限，它应该在测试的时候对着我的这颗代码它一边测一边修改说我测不过，我把那儿修改一下就好了。然后它一直在 Hack 这种法子对吧。所以说这里面会有很多的这种全线的治理，包括上层的这种协调关系的编排的一些问题。这属于第三层。

所以大家过去做一个 Agent 的时候说到的这三层都是要靠手搓的，但现在慢慢的可能有一些已经封装成了一些可以直接调的 Infra 工具了对吧。

### KB 的 K 系列 Agent 工具链

再你看了现在有哪些已经封装的很好的，有哪些人家目前需要手搓，这三层到目前我并没有觉得有封装的非常好的，因为我们觉得这个范图变化太快了。从 22 年底 ChatGPT 的发布，再到现在类似于 CC 这一套背后，这种 Agent 的智能体模型的这套范图，模型被训练的完成长程任务，也就是最近半年左右才开始出现。所以开源社区我其实也观察了很多，就是没有看到太让我满意的这一层。所以也是我们自己在做这一层的原因。要不这个也请你介绍一下，你们在做的这一层具体是什么？这是你们创业的一个项目对吧。

新璐：我们在做这一层。我要刚刚说的 Agent 的三个层面，我们去提供了一个完整的工具链。

首先我们最底层是一个叫 Komputer 的工具链，就是它是取 Computer 的 C 的那个字母变成 K，叫 Komputer。它是一个我们在数据结构上实现的一台 Unix 计算机，就是你可以理解为我们在内存中实现的一台虚拟的计算机。它给我们像小龙虾，包括 Claude Code 等这种，下一代的一种主动式的 Agent，它有它的自底带性成长性，我们要给这种 Agent 配一个它的执行环境，就是让它在那个环境里去生活和工作。然后这是我们提出的我们认为 Unix 的计算机可能就是对它最好的居住和生活的环境。我们做的第一层是这个对。

然后我们上层会有一个叫 K Runtime 的东西，它是一个 Agent Runtime 这一层，我们会相当于提供给开发者去开发 Agent 的对象的一些接口，语法堂上的一个缝装。除了这一层我们还会有其他很多跟大家正交的层面，比如说像 K Watch 对吧，这是我的观测。比如说你过去半个月我的 Agent 他们分别在任务完成在什么情况下遇到卡点对吧，它这样就可以知道我去观测这不是我的 Skill 有设计好，还是我的 Scale 没有提供到位，还是说我用的模型不太行 OK。一个时候观测层就很重要了。以及我们基于观测层的话可以对数据做导出，导出以后我们会有一个叫 KRL 的一个工具链，你可以把这个数据拿来强化学习去训练你的模型，或者说你说我不想练我自己的模型，我就想用一个标准的模型的 API 去请求那个计划，我们也可以对你做这个上下文层面，比如说关于 Memory、关于 Skills、关于过去的一些经验的一提取和自底带的一个，就是你可以理解为成功人版的强化学习。

### vs. AWS AgentCore、阿里云 AgentBay

我理解其实云服务厂商也会想要做这一层，AWS 有 Agent Core，然后阿里云有 Agent Bay，也都是想为搭建 Agent 的人提供 Harness 这一层的 infra。然后你们作为创业公司会怎么看，你们和云服务厂商之间提供的这种差异？

新璐：首先就是我想强调我们的 Agent 开发工具链并不是为了完全运行在云上，或者放给云建的。其实我们是希望一切能在 C 端前都可以跑得起来。比如说你在开发一个浏览器里的网页，就是完全运行在浏览器上，网页没有后端的那种。然后包括你可能在开发一个微信小程序对吧，然后你开发很多场景下呢，它是塞不进一个 Linux 的对吧，你的微信小程序一个 package 可能也才几兆的大小。我们希望就是说在所有的能跑 JS 的场景，无差别的给你提供一个类似于上个世代 NodeJS 的那层的一个工具链空间，你就 Import 它就直接用好了。然后它背后的性能占用也非常极致，就是只有 1KB 那么大小的一个 Unix Komputer。然后可以你的 Agent 去用一个生活居住的环境。但是你的那个像小龙虾这种 Agent 在那个环境里放着，它又以为自己是生活在一个 Unix 的计算机上，它的命令行，它的文件系统，它的网络能力都是跟它平常训练的时候的新制是一致的。

所以你们的公司名字叫 KKB？就是 KB 的一个展开的线对。

对，这怎么做到的呢？这是因为我们是用数据结构重新实现 Unix 这件事。你可以理解为我们 Unix 的文件系统我们会用帖子重写成虚拟的 bass。然后再包括我们说那 OK 我是不是还除了前台进程，我有一个后台进程的一个运行能力 OK。我有我的 Cloud 中的概念，就在那个虚拟世界上往前推进。然后这一层我们要去做，包括我说要有居云网的组网能力，我要就类似 NAS，那这些很多方面能力我们都会在我们的虚拟抽象层用语言去重写一遍。对，然后包括在一些支持 WebAssembly 的场景，我们又会把一些我们的工具链用 WASM 重写一遍。然后在它能够支持的情况下我们会切换过去。然后它不止情况下我们会 fold back 到最普通的 JS 是我们的执行。

### Harness 执行能力层

好，我们还是再说回来 Agent Harness。我们说第一层是执行能力层，是要给 Agent 各种各样的工具。可以列举一下现在最主要的工具有哪些吗？

新璐：我觉得最主要的工具首先就是关于文件系统的那些工具，就是增删读写搜索这些大本益，真的不管是在做 Coding 任务还是做 Editor 之类的任务，基本上都会需要的一些工具。那第二层呢？我认为就是 Browser 这一层，就是你给它一个关于用户世界系统这样的一个操作能力。到这一层，第三层呢？其实有点像 Python、Node 这种的就是一个语言解释纪层。我觉得这是大部分情况下你配好这些工具，它应该就是可以满足我们可能 95% 以上的 Agent 任务了。

听起来其实配工具这一层很简单，但是这一层有你认为比较容易采的坑吗？

新璐：它很多时候跟我们的全线包括 Agent 的角色是相关的。比如说我某一个 Agent 就是负责去对这个代码库做一遍 Explorer 对吧。然后这个时候我应该只给它配就是没有副作用的工具。比如说所有关于写的命令，就是包括删啊改的能力，包括有一些你是不能让它操作 Browser 或者就是你配给它以后你是限制它访问很多网域的，就是这样的话会有很多的中途的 Tradeoff。就是你的 Agent 的工具链的设计要紧密的跟 Agent 的角色绑定去 Limit 它调它的底层对这台 Unix 的计算机的操作能力。

### Memory 的流派

那我们刚才说到这个第二层上下文语状态层这环境层的时候，Memory 就记忆是非常重要的。然后现在也可以看到有不同的记忆，不管是这个模型厂原生也有一些记忆的功能对吧，然后也有三方的一些开源的一些插件，它会给记忆的解决方案。然后可不可以给大家讲一讲，现在你认为 Agent Harness 的 Memory 这一层有哪些就是主要的解决方案？Memory 这一层的发展呢？

新璐：目前还处于一个很早期的阶段。我其实把它处理的分为就是规则式的、半规则式的和完全模型驱动式的。那目前落在使用上来看的话，相对规则式的和半规则式的我也多吧。

完全规则式的就是像很多基于这种知识图谱和向量搜索，他们所构成的就是把最后你的这个信息在最后抽象中不同节点的表示，然后他们之间在做关联，然后检索和扩展。那这种方式是非常结构化信息规则式做法的。当然我个人其实不太喜欢这种做法。然后我更喜欢的是像后者这种半规则式的，就是可能我底层是一个 Unix 的 Files，我就是通过大量的 Markdown 的方式来存出这种资料信息。其实 Claude Code 的也是这样做的，省下里面也是这样做的。它可以通过一些类似其他的空间关系，比如说像迷宫中这种走势，这是最近生化危机的那个女主最近做的特别红的那个记忆项目，这种其实就是底层找一种大家都公认的非常清晰的稳当健壮的假的方式，我有房间和空间位置关系，就是人也很容易懂，这个 LM 也很容易懂的一种底层的就是半结构化的这种方式的信息。然后它的更新也是半规则式的，就基本上是通过 Agent 来做一个更新，而不是完全 Rubis 的，一块一下就最后得到一个什么信息，再在知识图上做一个什么样的推理。我觉得那个是叫完全规则式、完全 Rubis 的、半规则式的做法其实就是用 Agent 来在这里边去，你把这个流程的规则告诉 Agent，但是整个分析和过程是由 Agent 去跑的。我现在要找什么信息完全是一个后台的 Agent，可能用一个更便宜的小模型去跑一遍。那更新其实也是这样的。像 Claude 过程中也有这种类似于做梦的机制，就是隔一天让它跑一下最近几个 Session 的对话，它提取下有用信息是可以对过去的记忆做这种更新、纠正和合并的。然后包括这一块有一个开源项目叫 MEMU 也做得很好，MEMU 对 MEMU 是一个开源项目，它其实也非常去拥抱 Unix Files 加上 Agent 去驱动维护更新和查询的整个方式。其实 Claude Code 里边也是这样设计的。感谢你大家可以去看一下我们自己的 Learn Claude Code 的开源仓库，MEMU 有这个开源仓库。
